# koe デザインブリーフ — 没入・存在感型(immersive orb) (2026-06-09)

> ⚠️ **SUPERSEDED (2026-06-10)**: この「没入 orb 一本」方針はユーザー判断で撤回。現行の正は
> **`2026-06-10-glassbox-console-design-brief.md`(見える glass-box コンソール + 音声主役)**。
> 本書の orb 状態表現/モーション/配色 craft は新方針の「音声状態インジケータ(縮小 orb)」へ流用するため保存。
> 経緯 = bd `koe-ios` ピボットノート。

ユーザー方針(2026-06-09): 現状の「ダーク管制盤(operator console)」路線は求める方向と違う → **全面リデザイン**。

## 確定した方向
- **没入・存在感型**: 画面中央に「呼吸する球(orb)/アンビエントな波形」を主役として大きく置き、
  AI がそこに「居る」感覚を作る。Siri / 音声アシスタント特化 UI に近い、静かで知的な没入感。
- **配色: OS 追従**(Windows/Mac のライト・ダーク設定に自動追従。両方デザインする)。
- 情報は orb 周りの最小オーバーレイ(ログをびっしり並べない)。
- 中心思想「校正された透明性(glass-box)」= AI が今なにを考え・実行し・どれくらい確信しているかを
  開示する「思考の窓」を orb 下に。これは装飾でなく koe の機能の主役。

## 進め方(2段階)
1. **Claude Design(アーティファクト)で生成**: 下記プロンプトを貼り、インタラクティブな UI プロトタイプを作る。
2. **koe React + CSS に実装**: 生成モックを `VoiceButton` / `ActivityLog` / `OnboardingGate` /
   `SettingsPanel` + 各 CSS に移植。Rust 駆動のイベント表示(session-status / tool-event /
   thinking-event / cost-update)に橋渡し(モックはそのままでは koe アーキに乗らない)。
   bd: `[epic] UI/UX 全面リデザイン`(label review-2026-06-09)。

## 維持すべき品質(リデザインで壊さない)
- コントラスト比 WCAG AA 以上(本文・CTA とも現状は AA 準拠)。
- anti-ai-smell: Inter 不使用 / purple→blue gradient 不使用 / border-radius・余白・モーション値を均一にしない。
- input は 16px(iOS 自動ズーム回避)。色だけに頼らずアイコン+文言で状態を伝える(色覚配慮)。
- prefers-reduced-motion を尊重(呼吸アニメの抑制版)。

## Claude Design に貼るプロンプト(全文)

```text
あなたは一流のプロダクトデザイナー兼フロントエンド実装者です。
下記のデスクトップ音声AIアプリ「koe」のUIを、実際に触れるインタラクティブな
プロトタイプとして作ってください。

【成果物】
- React + Tailwind の単一アーティファクト(または依存なしの単一HTMLファイル)。
- ライトテーマ / ダークテーマの両方。OSのライト・ダーク設定に自動追従する想定
  (プロトタイプ内に切り替えトグルも付ける)。
- 主要な「状態」を実際に切り替えて見られるデモにする(待機/接続中/会話中/作業中/
  再接続中/エラー、承認オーバーレイ、初回オンボーディング、設定パネル)。
- UIの言語は日本語。

──────────────────────────────────
■ 製品「koe」とは
──────────────────────────────────
- 起動しっぱなしのリアルタイム音声AI秘書。Windows/Mac のデスクトップアプリ。
- 人と話すように連続で会話しながら、裏でPC操作・Web検索・記録を実行する。
- 中心思想は「校正された透明性(glass-box)」:
  AIが "今なにを考え・なにを実行し・どれくらい確信しているか" を、画面と声で
  リアルタイムに開示する。即答ブラックボックスへのアンチテーゼ。人間が
  「いつ介入するか」を判断できることが価値。
  → だから「AIの状態」と「思考の窓」が、このUIの主役。装飾ではなく機能。

──────────────────────────────────
■ デザインの方向(必須・ここは外さない)
──────────────────────────────────
- 「没入・存在感」型。画面中央に "呼吸する球(orb)/アンビエントな波形" を
  主役として大きく置き、AIがそこに「居る」感覚を作る。Siri や音声アシスタント
  特化UIに近い、静かで知的な没入感。
- 情報は orb の周りに最小限のオーバーレイとして添える(ログをびっしり並べない)。
- 配色はOS追従(ライト/ダーク両方を必ずデザインする)。落ち着いた中立色を基調に、
  アクセントは1色だけ(彩度の高い色を全面に使わない)。
- ウィンドウは常駐アプリ。目安サイズ: 幅 440px × 高さ 680px の縦長(リサイズ可)。
  上 2/3 を orb の空間、下 1/3 を「思考の窓」とコントロールに使う構成を基本とする
  (より良い構成があれば提案可、ただし orb を主役にすること)。

──────────────────────────────────
■ orb の状態表現(一目で状態が伝わること)
──────────────────────────────────
中央の orb の動き・色・質感で次の状態を表現する:
- 待機 (idle):         ゆっくり静かに脈動。色は控えめ。「話しかけて」と小さく。
- 接続中/準備中:        立ち上がりのアニメーション(収束していく粒子など)。
- 会話中 (conversing): こちらの声・AIの声に反応して波打つ/揺らぐ。生きている感じ。
- 作業中 (working):    裏でツール実行中。回転や同心円パルスなど「動いている」表現。
- 再接続中:            会話中とは別の、リカバリ中とわかる控えめな動き。
- エラー:              赤系に沈静化し、短い日本語メッセージ+「再試行」を出す。

──────────────────────────────────
■ 「思考の窓」=最重要の差別化(透明性)
──────────────────────────────────
orb の下に、AIの内面を"検証可能な形だけ"開示する領域を作る。生の思考文(CoT)は
出さない。出すのは次の3つ:
1. 💭 いま考えていること: 短い要約1行(例:「10時の会議を確認しています」)
2. 実行中/直近のツールと参照ソース: 例 `web.search "新幹線 時刻"` / `calendar.read`
   (アイコン+ツール名+対象。完了は ✓、実行中はスピナー、失敗は ✕)
3. 校正済みの確信度ラベル: 3〜4段の自然言語ラベルで表示(例:「たしか / たぶん /
   要確認」)。生の％は出さない。控えめなバッジで添える。
   ※ 実際のラベル文言は今後の調整対象なので、デザイン上はプレースホルダでOK。

直近の動作は数件だけ薄く残す(履歴の全展開は別画面想定、ここでは出しすぎない)。

──────────────────────────────────
■ そのほかの画面・要素
──────────────────────────────────
1) コスト表示(常時、小さく):
   「今月 $0.42 / 上限 $20.00」のように使用額と上限を併記。上限に達したら
   「予算上限に達したため停止しました」+上限引き上げの入力を出す。

2) 承認オーバーレイ(危険操作の人間承認):
   コマンド実行・ファイル削除・外部送信など危険な操作の直前に、orb の前面に
   静かなオーバーレイで「◯◯を実行してもいいですか?」+[許可][拒否]。
   30秒で自動キャンセル(カウントダウン表示、無操作なら安全側=拒否)。

3) 初回オンボーディング(2ステップ):
   ① 月の予算上限を決める(金額入力/または「上限なし」を明示選択)
   ② 自分のOpenAI APIキーを入力(BYOK=キーは端末内に暗号保管、外部送信しない、
      の安心メッセージを添える)→ 完了。
   没入型の世界観の中で、最初に世界が立ち上がる演出だと良い。

4) 設定パネル:
   APIキー管理 / 音声の品質や提供元の選択 / 予算 / 操作を許可するフォルダ・URL。
   普段は orb 画面の隅の小さなアイコンから開く。

──────────────────────────────────
■ インタラクション・モーション
──────────────────────────────────
- orb のタップ=会話の開始/停止。キーボード操作(Space等)でも開始できること。
- モーションのイージングは用途で変える:
  呼吸=ゆったり、要素の入退場=減速/加速、ボタン操作=軽いスプリング。
  全部を同じ duration / 同じ ease にしない。
- prefers-reduced-motion を尊重(動きを抑えた版も成立させる)。

──────────────────────────────────
■ 品質ルール(重要・AI臭の回避とアクセシビリティ)
──────────────────────────────────
- フォントに Inter を使わない。purple→blue のグラデーションを使わない(AIテンプレ臭)。
  日本語が美しく出る font-family にする。
- 配色: 中立色2色+低彩度1色+アクセント1色程度。全色を同じ鮮やかさにしない。
- コントラスト比は WCAG AA 以上(本文テキストは十分に濃く/明るく)。
- border-radius・余白・アニメーション時間を全要素で均一にしない(リズムをつける)。
- 状態変化・エラー・承認は、色だけに頼らず文言とアイコンでも伝える(色覚配慮)。
- 文章は日本語として自然に。説明くさい定型文を避け、短く落ち着いたトーン。

──────────────────────────────────
■ 出力してほしいもの
──────────────────────────────────
1. メイン画面(待機/会話中/作業中/エラーを切り替えられるデモ)
2. 「思考の窓」が実際に動いて見える状態(ツール実行→完了の流れ)
3. 承認オーバーレイ
4. オンボーディング2ステップ
5. 設定パネル
6. ライト/ダーク両方
まず全体の世界観が伝わるメイン画面から作り、その後で各サブ画面を足してください。
```

## 調整余地(次セッションでユーザーに確認しても良い点)
- orb か波形か(または両立)。
- ウィンドウサイズ(440×680 は仮)。
- 確信度ラベルの具体文言(「たしか/たぶん/要確認」は仮、校正レイヤ koe-sua.2 で確定)。
